BRD · Business Requirements Document

Business Context & Requirements

Defining the organizational need, stakeholder expectations, and measurable business outcomes that this ML system must deliver.

Executive Problem Statement

A global travel platform loses significant booking revenue because new user homepages display generic content. The platform cannot predict where a new user wants to travel until they actively search — missing the critical first-impression personalization window. This cold-start problem means the platform treats its first 60 seconds with every new user as a blank slate, when in fact their demographic profile and early click behavior already carry predictive signal.

Business Opportunity

Browse-to-book conversion is the single most important funnel metric in the travel industry. A system that predicts a user's first booking destination within their first session — before any explicit search — enables real-time homepage personalization, targeted email campaigns within hours of signup, smarter ad re-targeting, and inventory pre-caching. Even a 2% conversion improvement on a platform processing $1.5B in annual bookings generates $30M in incremental revenue.

Business & Technical Challenges

The Four Core Problem Dimensions

Each challenge below represents a distinct failure mode that must be addressed for the system to deliver business value. Ignoring any one of them produces a model that looks good in notebooks but fails in production.

CHALLENGE 01

Cold-Start Problem

At the moment a user lands on the platform, the system knows almost nothing about them. There is no booking history, no stated preferences, and potentially only seconds of click behavior. Standard collaborative filtering and recommendation systems completely fail here because they require existing user-item interaction history to compute similarity.

Affects 100% of new users at activation moment
Mitigation: Demographic-first model + behavioral augmentation. Dual-model architecture: one for users with sessions, one for zero-session users. Graceful degradation guaranteed.
CHALLENGE 02

Severe Class Imbalance

The target variable has 12 classes. "NDF" (no booking) accounts for ~60% of all users. "US" accounts for ~30%. The remaining 10 international destinations share only 10% of observations. A naïve model that always predicts NDF or US achieves 90% accuracy — while delivering zero business value for personalization targeting international destinations.

10 destination classes have <1% representation each
Mitigation: SMOTE / class-weight balancing, NDCG@5 as primary metric (not accuracy), ensemble methods with per-class calibration, oversampling minority destinations.
CHALLENGE 03

Aggregation Design Complexity

The sessions table has 10M+ rows of raw clickstream events. One user may have 500 action records; another may have 3. The model requires a single fixed-length feature vector per user. How you collapse session rows into features determines the ceiling of your model. Average time-on-page? Action sequence TF-IDF? Last-action weighting? These design decisions are where the real ML work happens, yet there are no labels to guide them — only domain intuition and offline experiments.

Feature engineering drives 80% of model performance
Mitigation: Systematic aggregation grid (count / mean / max / last / TF-IDF), recency weighting (exponential decay), action-type hierarchies, sequence embedding via action "documents."
CHALLENGE 04

Text-as-Structured-Signal

Device type strings, action names, action types, and action details in the sessions table are categorical string fields that carry rich implicit signal. A user browsing on a recent iPhone in English from San Francisco has a fundamentally different booking profile from one on an Android in Portuguese from São Paulo. These strings need cleaning, clustering, and feature extraction — but they are not natural language, they are coded labels, requiring a hybrid NLP + categorical approach.

Device + language signals correlate with booking destination
Mitigation: Label encoding + frequency encoding for high-cardinality fields, action sequence treated as a "document" with TF-IDF / Word2Vec, device proxy features (OS recency, screen class), language as geo-proxy.

Model Requirements

  • Multi-class classification (12 classes) with calibrated probability outputs
  • Dual-model architecture: full model (with sessions) + fallback model (demographics only)
  • Primary algorithm: XGBoost / LightGBM ensemble (gradient boosted trees proven optimal for tabular + imbalanced multi-class)
  • Secondary experiment: Random Forest baseline for comparison
  • Class weighting: inverse frequency weighting applied to all models
  • Evaluation metric: NDCG@5 (Normalized Discounted Cumulative Gain at rank 5)
  • Cross-validation: Stratified 5-fold (preserves class distribution per fold)
  • Hyperparameter optimization: Optuna Bayesian search, 100 trials
  • SHAP integration for feature importance and per-prediction explainability

Text Intelligence Pipeline (NLP Layer)

Step 1 · Cleaning
  • Lowercase + strip whitespace on all string columns
  • Unify device type variants (e.g. "iPad", "ipad", "IPAD")
  • Map null action_detail to "none" sentinel
  • Remove action records with secs_elapsed < 0 (invalid)
Step 2 · Sequence Encoding
  • Concatenate per-user action sequences into "document" strings
  • Fit TF-IDF on training corpus, top 50 components
  • Experiment: Word2Vec on action sequences (action2vec)
  • Recency-weight: recent actions scored 2× earlier actions
Step 3 · Signal Extraction
  • Device type → OS family + screen class proxy features
  • Language → geo-region proxy (16 language groups)
  • Action hierarchy: booking actions weighted 3× browse
  • Last-N actions: final 5 actions as separate categorical features

Project Roadmap

Sprint Plan & Deliverables

Two-sprint agile engagement structure. Sprint 1 focuses on discovery and baseline; Sprint 2 on engineering maturity and production readiness.

SPRINT 1 · Days 1–5 · Discovery & Foundation
Day 1–2

Data Ingestion & EDA

Profile all four tables. Document data quality issues. Produce class distribution charts. Identify zero-session user proportion. Commit initial findings notebook to GitHub.

Day 2–3

HLD + LLD Documents

Architecture diagram (HLD). Feature catalogue with types and imputation strategy (LLD). Metric selection justification. Committed to /docs before any model training begins.

Day 3–4

Baseline Feature Pipeline

Demographic features encoded. Basic session aggregates (count, total time). Train/test split with stratification. Baseline XGBoost trained and evaluated with NDCG@5.

Day 5

Sprint 1 Checkpoint Presentation

Data insights, architecture diagram, baseline model results, blockers identified, Sprint 2 plan confirmed.

SPRINT 2 · Days 6–10 · Engineering & Production
Day 6–7

Advanced Feature Engineering

TF-IDF on action sequences. Recency-weighted aggregates. Device proxy features. Action hierarchy weighting. Cold-start fallback model trained. Feature importance analysis with SHAP.

Day 7–8

Model Experimentation & Comparison

LightGBM vs XGBoost comparison. Class imbalance strategies (SMOTE vs class weights). Optuna hyperparameter search. Ensemble experiments. MLflow experiment tracking.

Day 8–9

Pipeline Modularization

Refactor notebooks into modular Python pipeline. Feature store serialization. Inference pipeline with latency benchmarking. Unit tests for key transforms. Docker packaging.

Day 10

Final Presentation & Business Translation

VP-level presentation: business context → data insights → technical approach → results → business recommendation → retrospective. All team members present.
